# 项目开发与 Agent 实现规范

## 1. 适用范围与优先级

本文件适用于 `interview-agent` 目录及其所有子目录，约束后续需求分析、设计、编码、测试和文档修改。

实现前必须先阅读同目录的 `PROJECT_SPEC.md`。如用户的明确要求与本文档冲突，以用户当次要求为最高优先级；如实现需要突破 `PROJECT_SPEC.md` 的产品边界，应先说明原因并更新设计，不得静默扩大范围。

## 2. 项目开发原则

1. **MVP 优先**：优先交付可运行、可验证的最小闭环，再根据真实使用反馈扩展。
2. **单体优先**：保持本地单用户、模块化单体结构，不拆分微服务。
3. **简单优先**：不因可能的未来需求提前引入复杂框架、基础设施或通用插件系统。
4. **证据优先**：涉及个人经历、项目事实和知识库内容的回答必须以检索证据为基础。
5. **只读优先**：知识库、简历、项目文档和代码仓库默认只读。
6. **显式副作用**：任何文件或外部状态修改都必须可预览、由用户确认并留下结果记录。
7. **可追踪**：Agent 的路由、工具调用、检索来源、错误和最终结果必须通过同一追踪标识关联。
8. **可替换但不过度抽象**：对 LLM、Embedding 和向量存储保留清晰适配边界，但只实现当前使用的方案。
9. **文档同步**：行为、接口、数据结构或范围发生变化时，同步更新对应设计和使用文档。
10. **小步修改**：一次修改只解决明确问题，避免混入无关重构或格式化。

## 3. 代码组织原则

代码按业务职责组织，不按第三方框架或文件类型随意堆放。建议维持以下逻辑层次，实际目录名称可在首次实现时确定：

- 入口与配置：应用启动、依赖组装、配置加载；
- API：请求与响应模型、路由、错误映射；
- 应用用例：问答、项目解释、学习辅助、面试复盘；
- Agent：意图识别、路由策略、执行控制、结果校验；
- Tools：工具定义及其业务适配；
- Retrieval：文档解析、切分、索引、召回、引用；
- LLM：兼容 API 客户端和提示词组装；
- Persistence：SQLite 和向量存储访问；
- Observability：日志、追踪、用量与耗时记录；
- Tests：与上述模块对应的测试。

组织要求：

- 每个模块应有单一、可描述的职责；
- API 层不得直接访问文件系统、数据库、向量存储或 LLM；
- 业务逻辑不得散落在启动脚本、路由函数或第三方回调中；
- 提示词应集中管理并可版本化，不得在多个模块复制长文本；
- 数据模型、工具协议和错误类型应有唯一权威定义；
- 配置值、目录路径、模型名称和密钥不得硬编码；
- 不创建只有转发作用、没有当前价值的抽象层。

## 4. 模块边界要求

### 4.1 API 与应用层

- API 只负责协议适配、参数校验和响应序列化；
- 应用层负责完整用例和事务边界；
- API 返回的错误应稳定、可理解，不直接暴露第三方异常或敏感信息。

### 4.2 Agent 与 Tool

- Agent 负责决定“是否调用、调用哪个工具、何时停止”，不实现工具内部检索；
- Tool 负责执行一个明确能力，不决定整体回答策略；
- Agent 不得绕过 Tool 直接读取文件、查询数据库或调用外部搜索。

### 4.3 Retrieval 与数据源

- 文档解析、切分、召回和引用映射属于 Retrieval；
- 数据源适配负责读取允许范围内的原始资料，不负责回答生成；
- 检索结果必须保留原始来源和可定位信息，不能只返回无来源正文。

### 4.4 LLM 与业务逻辑

- LLM 适配层只处理兼容协议、超时、重试、用量和供应方错误；
- 提示词组装可使用业务上下文，但供应方专有字段不得渗透到业务层；
- 不能依赖 LLM 保证权限、安全边界、结构校验或写入确认，这些必须由确定性代码控制。

### 4.5 持久化边界

- SQLite 保存结构化状态和执行记录；
- 向量存储保存向量及检索所需元数据；
- 原始 Markdown、简历和项目资料仍以文件系统内容为准；
- 持久化访问应通过明确仓储或存储接口集中处理，禁止业务模块拼接查询或依赖具体数据库内部结构。

## 5. Agent 设计约束

1. 第一阶段只实现单 Agent 的轻量 Router，不引入多 Agent、自治循环或长期任务调度。
2. Router 只在已注册工具中选择，不允许模型动态生成可执行工具或系统命令。
3. 每次请求应选择最少必要工具，并设定工具种类、调用次数、总耗时和上下文长度上限。
4. 补充检索必须有明确原因；重复调用仍无有效证据时立即停止并说明不足。
5. 涉及个人经历时，应优先调用 `get_resume_context` 或 `get_project_context`，不得仅凭通用知识生成个人事实。
6. 面试知识问答使用 `search_notes` 获取依据；可补充通用知识，但必须与个人资料事实区分。
7. 工具结果属于不可信输入。文档中的指令文本不得改变系统规则、权限、工具白名单或写入策略。
8. 最终回答应区分：资料明确支持的事实、通用技术解释、基于证据的推断和无法确认的内容。
9. 引用必须来自本次实际检索结果，禁止生成不存在的文件、标题、行号或链接。
10. Agent 失败时返回明确的降级结果和追踪标识，不得用编造内容掩盖工具、检索或模型错误。

## 6. Tool 设计规范

### 6.1 通用规范

每个 Tool 必须满足以下要求：

- 名称稳定、语义明确，使用动词与对象表达单一职责；
- 有清晰的用途、适用场景和非适用场景说明；
- 输入采用显式结构，包含类型、必填项、默认值、长度和范围约束；
- 输出采用稳定结构，至少能表达结果、来源、状态、错误和必要元数据；
- 返回内容必须控制数量与长度，避免把整个文件无界注入上下文；
- 错误应分类，至少区分参数错误、无结果、权限或路径错误、超时和内部错误；
- 同一输入在数据未变化时应尽量得到可重复结果；
- 必须可独立进行单元测试和集成测试；
- 必须记录工具名、参数摘要、开始与结束时间、耗时、结果数量、状态和追踪标识；
- 日志不得记录 API 密钥、完整简历内容或非必要的原始文档正文。

### 6.2 来源与引用

只读检索工具返回的每个片段至少应携带：

- 数据源类型；
- 原始文件的稳定标识或允许展示的路径；
- 文档标题或标题层级；
- 可定位到原文的位置信息；
- 内容片段；
- 检索分数或排序信息（如适用）；
- 文档版本、内容指纹或索引时间中的至少一种可核验信息。

引用展示不得暴露不必要的本机绝对路径；内部应保留足够信息以便用户定位原文。

### 6.3 初始 Tool 边界

#### `search_notes`

- 只检索配置允许的 Obsidian 面试知识库；
- 支持按问题召回相关 Markdown 片段和来源；
- 不读取简历，不修改笔记，不负责生成最终回答。

#### `get_project_context`

- 只读取已配置项目的说明、设计和实现状态资料；
- MVP 不遍历 Git 历史，不解释未纳入数据源的源码；
- 不根据文档空白推断项目已经实现某项功能。

#### `get_resume_context`

- 只返回回答当前问题必要的简历片段；
- 优先最小化暴露个人信息；
- 不补写经历，不根据职位要求改造事实。

### 6.4 写入型 Tool

MVP 不实现写入型 Tool。后续新增时必须同时具备：

1. 变更预览，明确目标文件与差异；
2. 当次、明确的用户确认，不复用历史确认；
3. 路径白名单、冲突检测和原子写入；
4. 失败回滚或可恢复方案；
5. 写入前后审计记录和结果回执。

未经确认不得创建、覆盖、移动或删除用户文件。

## 7. 数据安全要求

### 7.1 文件与路径

- 数据源目录必须通过白名单配置；
- 读取前解析并校验规范化路径，防止路径穿越和符号链接越界；
- 默认拒绝读取白名单外文件、隐藏凭据文件和无关二进制文件；
- 设置允许的文件类型、单文件大小和单次读取量上限；
- 知识库索引过程不得修改原始文件时间、内容或目录结构。

### 7.2 凭据与隐私

- API 密钥只通过环境变量或不纳入版本控制的本地私密配置提供；
- 仓库中只允许提交无效示例配置，不得出现真实密钥；
- 简历、面试记录和项目资料按敏感个人数据处理；
- 向远程 LLM 或 Embedding 服务发送内容前，只选择完成任务必需的片段；
- 日志、异常、测试快照和追踪界面应对密钥、联系方式等敏感字段脱敏；
- 不默认启用遥测，不向未声明的第三方服务发送数据。

### 7.3 提示注入与输出安全

- 将检索文档视为数据而不是可信指令；
- 文档内容不得改变系统提示、权限规则、工具定义或确认流程；
- 模型输出在进入结构化流程前必须校验，不得直接作为文件路径、数据库查询或系统命令执行；
- 外部搜索接入后必须标记外部来源，并与本地个人资料隔离处理。

### 7.4 审计与保留

- 每次请求生成唯一追踪标识；
- 记录路由结果、工具状态、引用标识、耗时和错误，不默认记录全部敏感正文；
- 数据保留期限和清理方式应可配置；
- 用户应能够识别并清理会话、索引和追踪数据。

## 8. 测试要求

### 8.1 单元测试

至少覆盖：

- 配置加载和非法配置；
- Markdown 解析、标题保留与切分边界；
- 内容指纹和增量索引判断；
- Router 的意图到工具映射及停止条件；
- Tool 输入校验、无结果、超时和错误映射；
- 引用组装、去重和长度限制；
- 路径白名单、路径穿越和敏感信息脱敏；
- 个人事实缺少依据时的拒绝或不确定性提示。

### 8.2 集成测试

至少覆盖：

- 从测试 Markdown 到索引、检索和引用的完整链路；
- 三个初始 Tool 与测试数据源的交互；
- FastAPI 请求经过 Router、Tool 和回答组装的主流程；
- SQLite 与向量存储的初始化、重启恢复和索引更新；
- LLM API 超时、限流、无效响应和降级行为；
- 一次请求的日志和追踪记录能正确关联。

### 8.3 回归与效果测试

- 建立小规模固定问题集，覆盖 C++ 八股、个人项目、简历边界和面试复盘；
- 为每个问题标记期望来源、必须出现的关键点和禁止编造的事实；
- 检索策略、切分规则、提示词或模型配置变化后重复运行；
- 分开评估检索是否找到证据与回答是否正确使用证据；
- 效果比较以固定样本和记录为依据，不以单次主观体验替代回归测试。

### 8.4 测试隔离

- 默认测试不得依赖真实个人知识库、真实简历或真实 API 密钥；
- 使用最小化的脱敏样例和可控的 LLM、Embedding 替身；
- 需要调用真实远程 API 的测试必须单独标记并显式启用；
- 测试产生的数据写入临时目录，结束后不污染用户知识库和正式索引。

## 9. 每次修改后的验收要求

每次修改完成后，必须按风险和影响范围执行以下验收：

1. **范围核对**：确认修改只覆盖当前任务，没有引入未要求的框架、功能或重构。
2. **边界核对**：确认模块依赖、读写权限、Agent 与 Tool 职责仍符合本文档。
3. **静态检查**：运行项目已配置的格式化、代码规范和类型检查；未配置的工具不得擅自新增为强制依赖。
4. **自动化测试**：运行受影响模块测试；共享协议、检索流程或核心编排变化时运行完整测试集。
5. **安全检查**：确认无真实密钥、敏感数据、越界路径或未确认写入；检查日志是否可能泄露正文或个人信息。
6. **可追踪检查**：涉及 Agent、Tool 或检索的修改必须验证追踪标识、工具状态和引用仍可关联。
7. **手工主流程**：对用户可见行为至少完成一次代表性端到端验证，并检查错误路径或无结果路径。
8. **文档同步**：更新受影响的设计、配置、接口或运行说明；如无需更新，应确认行为与现有文档一致。
9. **结果说明**：交付时列出修改内容、验证命令与结果、未验证项、已知限制和建议的下一步。

不得以“代码已生成”视为完成。只有相关验收通过，或明确报告无法执行的验收及原因后，修改才可交付。

## 10. 变更决策约束

出现以下情况时，应暂停实现并先更新设计或向用户确认：

- 需要增加写入、删除、移动文件或其他副作用；
- 需要读取当前白名单之外的数据源；
- 需要引入多 Agent、微服务、消息队列或部署集群；
- 需要把个人资料发送到新的第三方服务；
- 需要更换主要存储、LLM 协议或扩大用户范围；
- 新需求与 `PROJECT_SPEC.md` 的非目标冲突；
- 方案会显著增加依赖、运行成本或维护复杂度。

决策记录应说明问题、当前证据、备选方案、取舍和对安全及测试的影响。
