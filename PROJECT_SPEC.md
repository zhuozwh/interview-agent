# 个人技术知识与面试 Agent 项目设计规范

## 1. 文档目的

本文档定义“个人技术知识与面试 Agent”的产品边界、总体架构与阶段性建设路线，作为后续需求拆分、技术设计、实现和验收的统一依据。

项目优先解决个人秋招准备中的真实问题，以可用、可验证、可持续迭代为目标。若后续实现与本文档冲突，应先更新并评审本文档，再修改代码。

## 2. 项目定位

本项目是一个本地运行、面向个人使用的垂直领域 AI Agent 应用，服务对象是准备 C++ 后端岗位、同时希望体现 AI Agent 应用开发能力的候选人。

项目基于现有的 OpenAI-compatible LLM API、个人知识库和项目资料进行检索、编排与回答，不研究或训练基础模型，也不定位为通用聊天机器人。

核心价值包括：

- 将分散在 Obsidian、简历和项目文档中的信息组织为可检索上下文；
- 生成贴合个人真实经历的面试回答和追问；
- 用引用来源和执行轨迹约束回答，降低幻觉；
- 形成从学习、模拟面试到面试复盘的个人闭环；
- 以清晰、克制的工程实现展示 Agent、RAG 和后端开发能力。

## 3. 产品目标

### 3.1 主要目标

1. 对面试问题生成结构清晰、证据可追溯且不超出个人经历的回答。
2. 基于个人项目资料解释模块设计、实现状态和关键技术选择。
3. 结合当前上下文辅助技术学习，避免每次重新补充背景。
4. 整理面试记录，识别知识短板，并给出可执行的复习建议。
5. 保证知识库默认只读，所有工具调用均可查看、可审计。

### 3.2 成功标准

- 回答能够标注实际使用的知识来源；
- 涉及个人经历时优先依据简历和项目资料，不擅自补全事实；
- 缺少证据时明确说明不确定性，并提示用户补充资料；
- 一次请求的意图、工具调用、检索结果和最终回答可以关联追踪；
- 本地单机环境可以稳定完成知识库索引、检索和问答；
- 后续新增数据源或工具时，不需要重写核心编排流程。

## 4. 用户场景

### 4.1 面试问题回答

用户输入一个 C++ 后端相关问题，系统从八股笔记、历史面经、项目资料和简历中检索相关内容，生成适合口头表达的回答，并给出可能的面试官追问。

回答应区分通用技术知识与个人经历。涉及“你做过什么”“为什么这样设计”等问题时，必须以真实资料为依据。

### 4.2 项目理解辅助

用户询问个人 C++ 服务框架的模块职责、设计取舍或当前实现状态，系统读取项目说明和设计文档后进行解释。

MVP 只处理已经纳入知识源的项目文档；Git 仓库读取、源码搜索和逐行分析属于后续能力。MATLAB 项目的接入也在后续阶段进行。

### 4.3 技术学习辅助

用户围绕某个项目模块或技术主题继续追问，系统结合当前会话和检索上下文进行分层讲解，可说明概念、关联项目实践，并指出仍需查证的内容。

### 4.4 面试复盘

用户提供已有面试记录，系统按问题、回答表现、暴露短板和后续行动进行归纳。MVP 输出复盘结果但不自动回写 Obsidian；需要保存时由用户手动操作。自动或辅助写入属于后续扩展，且必须逐次确认。

## 5. 最终产品形态

最终产品是一个本地单用户应用，主体为单体 Python 服务，提供以下能力：

- 通过简单交互界面或 API 提交问题；
- 管理本地 Markdown、简历和项目资料的索引状态；
- 展示回答、引用来源、置信提示和可能追问；
- 查看每次请求的 Agent 路由、工具调用和错误信息；
- 在明确确认后执行受控的知识库写入；
- 按阶段扩展 Git 源码、MATLAB 项目和外部搜索等数据源。

FastAPI 自动生成的接口文档可作为早期调试入口。是否增加简单网页界面由后续实际使用反馈决定，不在 MVP 中建设复杂前端。

## 6. MVP 范围

### 6.1 MVP 必须包含

- 本地启动的 FastAPI 单体服务；
- OpenAI-compatible API 接入，首选 DeepSeek API；
- LLM 服务配置、超时、错误处理和基础重试；
- Markdown 文档扫描、解析、切分和增量索引；
- Embedding 与向量检索；
- Chroma 或 FAISS 中的一种作为首个向量存储实现；
- SQLite 保存文档元数据、会话信息、调用轨迹和索引状态；
- 轻量 Agent Router，根据请求选择必要工具；
- `search_notes`、`get_project_context`、`get_resume_context` 三个只读工具；
- 回答生成、来源引用、个人经历边界提示和可能追问；
- 面试记录的只读分析与复盘结果生成；
- 基础自动化测试和可重复的本地验收流程。

### 6.2 MVP 使用边界

- 默认仅索引配置允许范围内的文件；
- 原始文档仍由文件系统管理，数据库只保存必要元数据、索引和运行记录；
- 不自动修改 Obsidian、简历或项目资料；
- 不要求读取或理解完整代码仓库；
- 不承诺在知识来源不足时给出确定答案；
- 一次请求优先完成单一明确任务，不构建长链自主规划。

## 7. 后续扩展路线

### 阶段一：可用的知识问答 MVP

完成 Markdown RAG、三个初始工具、轻量路由、引用展示和调用追踪，覆盖基本面试问答、项目文档解释与复盘分析。

### 阶段二：检索与评测增强

- 优化切分、元数据过滤、召回和排序；
- 增加固定问题集与检索评测；
- 改善多轮上下文管理和引用一致性；
- 根据实际效果决定是否增加关键词与向量混合检索或重排序。

该阶段只在评测证明有必要时引入额外检索组件。

### 阶段三：代码与项目理解

- 接入 Git 代码仓库的只读扫描；
- 支持符号或文本搜索、文件级解释和逐步代码分析；
- 建立文档、代码和项目实现状态之间的关联；
- 在 C++ 项目能力稳定后，再评估 MATLAB 项目支持。

### 阶段四：受控写入与外部信息

- 增加经过用户确认的 Obsidian 写入；
- 支持复盘结果、待学习项等结构化落库；
- 按需增加外部搜索，并明确区分本地资料与外部来源；
- 对所有副作用操作增加预览、确认、结果回执和审计记录。

各阶段按真实需求推进，不以路线图为由提前引入复杂架构。

## 8. 总体架构

系统采用本地单体、分层模块化架构。模块之间通过明确接口协作，但部署时保持单进程或少量本地进程，不拆分微服务。

```text
交互层（FastAPI / 后续可选简单界面）
                │
应用层（请求、会话、问答、复盘用例）
                │
Agent 层（意图判断、轻量路由、执行控制）
          ┌─────┴─────┐
          │           │
工具层（只读工具）   回答生成与校验
          │           │
检索层（解析、切分、Embedding、召回、引用）
          │
数据源（Obsidian / 项目文档 / 简历）

基础设施层：LLM API、向量存储、SQLite、配置、日志与追踪
```

依赖方向应从交互层和应用层指向领域能力接口，再由基础设施实现具体适配。Agent 不直接操作文件系统、数据库或第三方 API，必须通过工具或专用服务完成。

## 9. 模块划分

### 9.1 API 与交互模块

负责请求校验、响应格式、错误映射和调用入口，不包含检索、提示词或路由逻辑。

### 9.2 应用用例模块

负责组织问答、项目解释、学习辅助和面试复盘等用例，管理会话上下文及一次请求的生命周期。

### 9.3 Agent Router 模块

负责识别请求类型、选择最少必要工具、控制调用次数和汇总工具结果。第一阶段不承担开放式长期规划，不引入多 Agent 协作。

### 9.4 Tool 模块

为 Agent 提供稳定、窄职责、可测试的能力边界：

- `search_notes`：检索 Obsidian 面试知识库，返回相关片段及来源；
- `get_project_context`：读取项目说明、设计文档和实现状态；
- `get_resume_context`：读取简历中的真实经历、技能和项目描述。

### 9.5 RAG 与索引模块

负责文档发现、格式解析、切分、Embedding、索引更新、向量召回和引用组装。切分结果必须保留到原始文件及位置的映射。

### 9.6 LLM 适配模块

封装 OpenAI-compatible API，统一模型配置、消息格式、超时、重试、用量统计和错误类型。业务模块不依赖 DeepSeek 的专用调用细节。

### 9.7 持久化模块

SQLite 用于保存结构化元数据、索引状态、会话摘要和执行轨迹；向量数据库用于保存 Embedding 及检索元数据。不得将二者职责混合为隐式共享状态。

### 9.8 配置、日志与可观测性模块

负责数据源白名单、模型参数、存储路径、日志级别和追踪标识。密钥通过环境变量或本地私密配置注入，不写入代码、文档索引或普通日志。

## 10. 数据流

### 10.1 文档索引数据流

1. 从配置的允许目录发现 Markdown、项目说明和简历资料。
2. 校验文件类型、路径和大小，忽略不在允许范围内的内容。
3. 解析正文与必要元数据，并按可配置策略切分。
4. 为每个片段保留数据源类型、文件路径、标题层级、位置和内容指纹。
5. 调用 Embedding 服务生成向量并写入向量存储。
6. 在 SQLite 中更新文档版本、内容指纹、索引时间和处理状态。
7. 文件未变化时跳过重复处理；删除或移动后同步清理失效索引。

### 10.2 问答数据流

1. API 接收问题和可选会话标识，生成请求追踪标识。
2. 应用层加载必要的会话上下文。
3. Agent Router 判断意图并选择最少必要工具。
4. 工具从知识库、项目资料或简历中返回结构化片段和来源。
5. 应用层对结果去重、截断并按来源类型组织上下文。
6. LLM 根据问题、会话上下文和证据生成回答与可能追问。
7. 回答校验检查引用映射、个人经历依据和不确定性表达。
8. 系统返回回答、引用、工具摘要和追踪标识，并记录本次执行轨迹。

### 10.3 面试复盘数据流

1. 用户提交或指定已有面试记录。
2. 系统提取问题、回答表现、未回答项和面试官反馈。
3. 按需检索知识库作为对照，不将推测当作实际面试事实。
4. 生成不足分析、优先级和后续学习建议。
5. MVP 只返回结果；如需保存，由用户手动处理。

## 11. Agent 执行流程

一次 Agent 请求按以下顺序执行：

1. **请求识别**：判断任务属于面试回答、项目解释、学习辅助、面试复盘或不支持类型。
2. **证据需求判断**：识别是否涉及个人经历、项目事实或知识库内容。
3. **工具选择**：只选择完成当前任务所必需的工具，避免无目的并行调用。
4. **工具执行**：校验参数、执行只读操作，记录输入摘要、耗时、状态和返回来源。
5. **上下文构建**：对工具结果进行去重、排序和长度控制，保留引用映射。
6. **回答生成**：将用户问题与证据交给 LLM，要求区分资料事实、通用知识和推断。
7. **结果校验**：检查关键个人事实是否有依据、引用是否存在、证据不足是否被明确说明。
8. **响应与追踪**：返回最终回答、可能追问、引用和追踪标识；内部保存完整调用轨迹。

默认限制单次执行的工具种类和调用次数。若第一次检索证据不足，可进行一次有明确目的的补充检索；仍不足时应停止扩展调用并向用户说明。

## 12. 技术选型说明

### 12.1 Python

适合快速构建 LLM API、文本处理和 RAG 流程，相关生态成熟，也便于以清晰模块展示应用层工程能力。

### 12.2 FastAPI

用于提供本地 HTTP API、参数校验和自动接口文档。MVP 不在其上建设复杂的服务治理或分布式能力。

### 12.3 SQLite

适合本地单用户场景，部署成本低，可承载文档元数据、会话信息、索引状态和调用轨迹。数据访问应集中封装，以便后续维护，但不为假设的分布式需求预留复杂抽象。

### 12.4 OpenAI-compatible API 与 DeepSeek

通过兼容协议降低模型供应方耦合，首选 DeepSeek API，不依赖 GPT API。具体模型名称、上下文长度和参数由配置管理，避免写死在业务逻辑中。

### 12.5 Markdown

Obsidian 知识库以 Markdown 为主，格式开放、便于版本管理和保留标题层级。首阶段优先保证 Markdown 解析和引用可靠性，不同时扩展大量文件格式。

### 12.6 Chroma 或 FAISS

二者均可满足本地 MVP 的向量检索需求。首个实现应在小规模技术验证后择一，选择标准包括本地部署、元数据过滤、持久化、调试便利性和维护成本。未完成验证前不同时维护两套正式实现。

### 12.7 轻量 Agent Router

项目只有少量明确工具，使用意图分类与受限路由即可满足 MVP。只有在真实场景证明固定路由不足时，才评估更复杂的规划框架。

## 13. 明确非目标

以下内容不属于当前项目目标：

- 训练、微调或评测基础大模型算法；
- 自研 Embedding 模型或向量数据库；
- 构建通用型自主 Agent 平台；
- 在初期引入多 Agent 系统；
- 拆分微服务或建设服务网格；
- 建设复杂前端、移动端或多用户权限系统；
- 引入 Kubernetes、Kafka 等与本地单用户场景不匹配的基础设施；
- 默认修改 Obsidian、简历、项目文档或代码仓库；
- 自动投递简历、自动参加面试或代替用户作出求职决策；
- 对知识库未覆盖的个人经历进行想象或补全；
- 为尚未出现的扩展需求提前搭建复杂插件系统。

任何新增需求都应先说明它解决的实际问题、是否进入当前阶段，以及对安全、数据和测试的影响。
